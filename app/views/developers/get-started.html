Get started

Developing your own template on Blot will be familiar to those who have developed templates for other platforms. You will need a working knowledge of HTML and CSS.

You can duplicate an existing template on the Template page of your blog’s dashboard. You can then make changes to the template using the online editor.

What a template contains

A Blot template is made of Mustache views and static assets. Work only on template behavior and presentation (HTML/CSS/JS and template config); do not modify Blot server code.

Typical template files:

  • package.json — Template configuration: locals (template-wide variables for all views), views (per-file routing via url and per-view partials). If you’re unsure which view maps to which URL, check views[*].url in package.json.
  • View files — e.g. entries.html (homepage), entry.html (single post), archives.html, plus optional search, RSS/Atom feed, or sitemap (names and routes defined in package.json).
  • style.css — Main stylesheet (plain CSS; no SCSS compilation).
  • script.js — Optional client-side behavior (keep it lightweight).

Template language and safety

Templates use Mustache:

  • Variables are HTML-escaped by default.
  • Use triple-braces {{{ ... }}} only for trusted HTML produced by Blot (e.g. rendered entry content).
  • Prefer sections and inverted sections instead of “logic.”
  • Partials: {{> header}} includes header.html (file partial, inherits context). Per-view string partials come from package.json under views[view].partials. You can embed rendered site files: {{> /Pages/Home.md}} includes the rendered HTML of that file.

Blot-specific constraints

  • No subdirectories for view files. Keep view files in the template root.
  • No SCSS/Sass compilation. Use plain .css only.
  • Blot routes templates before serving static files. Avoid a view whose url conflicts with an existing post or page path unless that’s intentional.
  • Avoid view-name collisions: don’t create two views with the same basename (e.g. feed.xml and feed.rss) if you can avoid it.

Dashboard controls via locals naming

When adding new locals in package.json, use established naming conventions so Blot can generate dashboard controls automatically:

  • *_color → color picker
  • font or *_font → font picker (except syntax_highlighter_font)
  • *_url → upload control
  • navigation_* / *_navigation → navigation options group
  • date_display + hide_dates → date format and toggle
  • syntax_highlighter (and optional syntax_highlighter_font) → syntax theme controls

Do not rename existing locals lightly; it can break existing template settings.

Edit templates locally

You can download a ZIP file of the template’s source files from the web template editor. After downloading and unzipping the file, move the extracted files into your blog’s folder under the following subdirectory:

/Templates/

Blot will automatically build the template from the files placed in this directory. However, the new local template will not be installed on your site automatically. Instead, it will appear in the list of templates available on the web template editor, where you can select it to use on your site.

Edit and test loop

On a hosted Blot site: Use ?debug=true (or ?json=true on some installs) on any page to inspect the exact JSON context used for rendering. Make a change, reload the page, and verify HTML output and CSS/JS asset URLs.

Local full-stack testing (optional): Run Blot locally and use the example site to render the template end-to-end. See the developer docs for Docker, npm start, and the https://example.localhost test site.

TIP: You can inspect the JSON data used to render a template if you append the query string ?debug=true to any URL on your blog.

Becoming proficient

After you have browsed the developer documentation, consider duplicating a copy of one of Blot's templates and reading its source code. Keep changes minimal and backwards-compatible, prefer semantic HTML and accessible patterns, and keep client JS optional so the site degrades gracefully if JS is disabled. When adding a new view or route, update package.json and link it from the template’s navigation where appropriate.
